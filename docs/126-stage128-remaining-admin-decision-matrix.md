# 阶段 128：剩余 4 个运维接口发布决策矩阵

## 背景

到阶段 127 为止：

- 桶、文件/对象主路径已经有真实 live 验证。
- 高频只读 Admin 视图已经有真实 live typed/raw 验证。
- 29 个破坏性 Admin 边界里，25 个已经有独立 lab 证据。

仍未完全放行的只剩 4 个：

1. `ADMIN_SERVER_UPDATE`
2. `ADMIN_SERVER_UPDATE_V2`
3. `ADMIN_SR_PEER_EDIT`
4. `ADMIN_SR_PEER_REMOVE`

这些接口不再属于“常用用户路径未验证”，而是发布策略与运维边界决策。

## 决策矩阵

| 接口 | 当前证据 | 风险类型 | 建议 |
| --- | --- | --- | --- |
| `ADMIN_SERVER_UPDATE` | 已证明 typed/raw 可达；官方 Docker 场景会明确拒绝原地升级 | 维护窗口 + 升级回滚风险 | **保留代码入口，但降级为高级/维护窗口接口** |
| `ADMIN_SERVER_UPDATE_V2` | 已证明 typed/raw 可达；官方 Docker 场景会明确拒绝原地升级 | 维护窗口 + 升级回滚风险 | **保留代码入口，但降级为高级/维护窗口接口** |
| `ADMIN_SR_PEER_EDIT` | 未做真实 peer 维护窗口操作；服务端语义是内部 peer 拓扑维护 | 内部拓扑/站点复制维护 | **不建议作为普通用户主文档能力暴露** |
| `ADMIN_SR_PEER_REMOVE` | 未做真实 peer 维护窗口操作；服务端语义是内部 peer 拓扑维护 | 内部拓扑/站点复制维护 | **不建议作为普通用户主文档能力暴露** |

## 更具体的发布建议

### 1. `ADMIN_SERVER_UPDATE` / `ADMIN_SERVER_UPDATE_V2`

建议：

- **代码层保留**，因为当前 SDK 目标仍包含与 MinIO 能力面的完整对齐。
- **产品层降级**，不要在“常用能力”“快速开始”“普通运维示例”里推荐。
- **文档层明确**：只有具备升级计划、镜像来源控制、回滚手册和维护窗口时才允许使用。

不建议把“Docker 中返回明确拒绝”解释成“升级能力已验证完成”。

### 2. `ADMIN_SR_PEER_EDIT` / `ADMIN_SR_PEER_REMOVE`

从 MinIO 服务端实现语义看，这两个更接近：

- 内部 peer 拓扑同步
- 站点复制维护窗口操作
- 集群级运维编排能力

建议：

- **代码层可保留**，维持路由/能力面对齐。
- **用户文档层隐藏或至少标注 internal/advanced**。
- **不要把它们和 `siteReplicationEdit` / `siteReplicationRemove` 这种更贴近用户配置语义的入口混为一谈**。

## 面向用户的推荐口径

如果要对外给出一句非常直接的发布口径，可以用：

> 桶、对象/文件和常见只读 Admin 路径已经过真实环境验证；剩余 4 个接口属于维护窗口或内部拓扑运维能力，默认不作为普通用户常用能力承诺。

## 结论

这 4 个接口不构成 SDK “核心可用性”阻塞；它们构成的是：

1. 是否对普通用户公开强调；
2. 是否在文档里降级为高级/内部接口；
3. 是否要求真实维护窗口证据后再宣称“可安全使用”。

因此，下一步不应再继续把它们和桶/文件/高频 Admin 主路径混在一起，而应该由发布决策单独处理。
